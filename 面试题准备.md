# DeepResearchAgent 面试题准备

这份文档按**高频度**和**难度**整理，覆盖项目深挖、八股文、系统设计三大类。
**建议：每个问题自己先口述一遍，确保能讲 2-3 分钟不卡壳。**

## 一、项目深挖（必问，占面试 60% 时间）

### 1. 请介绍你最得意的这个项目，解决了什么问题？
**答法框架（STAR）：**
- **Situation**：当前 LLM 存在信息滞后、幻觉问题，用户做深度调研时需要手动搜索、对比、整合，效率低。
- **Task**：我想做一个能自动完成"规划 -> 搜索 -> 分析 -> 报告"全流程的 Agent。
- **Action**：自研 ReAct 循环、自适应规划器、冲突检测、可信度评分、SSE 流式推送。
- **Result**：支持 3 种深度模式，能生成带引用来源和冲突提示的 Markdown 报告。

### 2. 为什么不用 LangChain / LlamaIndex，要自研 ReAct？
**答：**
- LangChain 是"黑盒"，封装了大量魔法，面试时会被质疑"你到底懂不懂底层"。
- 自研可以精准控制每一步的 Prompt、Token 消耗、错误处理。
- 我的项目核心就是展示**对 Agent 架构的深度理解**，而不是调包能力。
- 补充：LangChain 的 LCEL 在复杂流程中调试困难，自研状态机更清晰。

### 3. ReAct 的 Thought / Action / Observation 具体怎么设计的？
**答：**
- **Thought**：基于当前 WorkingMemory 的上下文，让 LLM 决定下一步策略（如"我应该搜索 AI Agent 记忆机制的最新论文"）。
- **Action**：调用搜索工具（SearchTool）或网页浏览工具（BrowserTool），输入是 Thought 生成的查询词。
- **Observation**：工具返回的结果（网页标题、摘要、正文），写入 WorkingMemory。
- 循环终止条件：执行完 Planner 生成的所有步骤。

### 4. 你的自适应规划是怎么实现的？怎么判断问题复杂度？
**答：**
- 用 LLM 做 Zero-shot 分类，Prompt 里给出 shallow/medium/deep 的定义和示例。
- shallow：单一事实查询（1-2 个来源即可）；medium：需要综合对比；deep：趋势判断、多维度分析。
- 分类后，Planner 生成对应步数的执行计划（3/5/7步），并控制每步的搜索次数。

### 5. 冲突检测是怎么做的？准确率如何？
**答：**
- 每次收集到新来源后，取最近两条让 LLM 对比是否存在事实冲突。
- Prompt 要求 LLM 输出 JSON 格式：`{conflict: true/false, claim_a, claim_b, analysis}`。
- 由于 LLM 判断存在不确定性，目前作为**提示性**功能，在报告中标注"以下信息存在不同说法"，而不是强行仲裁。
- 准确率：对明显冲突（如"A 说 X，B 说非 X"）判断较准，对 subtle 的语义差异可能漏检，这是已知局限。

### 6. 可信度评分的依据是什么？可靠吗？
**答：**
- 目前是**规则+启发式**：可信域名（arxiv, github, wikipedia）加分，HTTPS 加分，内容长度适中也加分。
- 局限：没有考虑作者权威性、发布时间、引用量等。
- 未来改进：可以接入 PageRank、域名权威度 API，或用 LLM 做更细粒度的内容质量评估。

### 7. WorkingMemory 怎么设计的？如果调研很长，上下文超出窗口怎么办？
**答：**
- WorkingMemory 存储 thoughts、sources、conflicts、current_plan。
- 给 LLM 的上下文是**压缩后**的：只保留最近 8-10 条 thoughts，早期内容做摘要。
- 最终报告生成时，用 `get_collected_contents()` 把所有来源正文拼起来，这时不放入 ReAct 循环的上下文，而是单独作为一次长文本生成任务的输入。
- 如果来源还是太长，可以进一步做 Map-Reduce：先让每个来源生成摘要，再基于摘要生成报告。

### 8. SSE 流式推送是怎么实现的？为什么不用 WebSocket？
**答：**
- 后端用 FastAPI 的 `StreamingResponse`，返回 `text/event-stream`。
- 用 Python async generator `yield` 实时推送进度更新。
- 选 SSE 不选 WebSocket 的原因：
  1. SSE 是单向推送（服务器->客户端），基于 HTTP，更简单；
  2. 自动重连、兼容性好（过防火墙/代理更友好）；
  3. 本场景不需要客户端高频发消息，WebSocket 双向通信是多余的。

### 9. 搜索用的是 DuckDuckGo，如果被封了怎么办？
**答：**
- 已做降级设计：优先 Tavily API（质量高、专为 AI 设计），未配置时自动降级到 DuckDuckGo。
- 未来可接入更多源：Google Custom Search、Bing API、Serper.dev、甚至自建爬虫集群。
- 也可以接入垂直搜索：arXiv API、GitHub API、维基百科 API。

### 10. 项目的性能瓶颈在哪里？怎么优化？
**答：**
- **瓶颈 1：LLM 调用延迟**。每次 ReAct 循环要调 3-4 次 LLM（Think + Act 中的提取 + 冲突检测 + 报告生成），串行执行很慢。
  - 优化：Think 和 搜索可以并行；冲突检测可以批量异步；报告生成用流式输出。
- **瓶颈 2：网页抓取**。某些网站响应慢或被封。
  - 优化：设置超时、并发抓取（asyncio.gather）、接入 Jina AI Reader / Firecrawl 等专业服务。
- **瓶颈 3：Token 消耗**。长报告生成消耗大量 Token。
  - 优化：Map-Reduce 摘要、分层生成（先大纲再填充内容）。

## 二、AI Agent / LLM 八股文

### 11. 什么是 ReAct？和 CoT、ToT 有什么区别？
**答：**
- **CoT（Chain of Thought）**：只在推理时生成中间步骤，不调用外部工具。
- **ReAct**：Reasoning + Acting，每一步推理后**调用工具**获取外部信息，再基于观察继续推理。
- **ToT（Tree of Thoughts）**：维护多个推理路径（树结构），用搜索算法（BFS/DFS）探索最优解，适合开放性问题。
- 我的项目用 ReAct 是因为调研任务有明确流程（搜索->分析->总结），不需要多路径探索。

### 12. RAG 的完整链路是什么？每个环节怎么优化？
**答（按链路展开）：**
1. **文档解析**：PDF/网页->文本。难点：表格、图文混排。优化：用专门的解析库（如 Unstructured、LlamaParse）。
2. **分块（Chunking）**：固定长度、按段落、语义分块。优化：重叠窗口（overlap）、递归分块、根据内容类型动态选择策略。
3. **Embedding**：选模型（BGE、M3E、OpenAI text-embedding-3）。优化：微调领域 Embedding。
4. **向量检索**：HNSW 索引、混合检索（BM25 + 向量）。优化：重排序（Rerank，如 bge-reranker）。
5. **查询重写**：HyDE（假设文档嵌入）、查询扩展、多查询检索。
6. **生成**：上下文压缩、引用溯源。

### 13. 什么是 Agent 的记忆？有哪些类型？
**答：**
- **短期记忆（Working Memory）**：当前对话/任务的上下文，容量有限。
- **长期记忆**：
  - 向量记忆：用向量数据库存储历史经验，语义检索召回。
  - 实体记忆：提取关键实体（人名、公司、概念）及其关系，类似知识图谱。
  - 反思记忆（Reflection）：Agent 定期总结自己的行为和教训，写入长期记忆。
- 我的项目目前只有短期 WorkingMemory，长期记忆是后续扩展点。

### 14. Function Calling 的原理是什么？
**答：**
- LLM 不是在调用函数，而是在**生成符合 JSON Schema 的结构化文本**。
- 流程：
  1. 把工具定义（name, description, parameters）写入 Prompt；
  2. LLM 决定用哪个工具，生成参数 JSON；
  3. 后端解析 JSON，执行真实函数；
  4. 把函数返回值塞回 Prompt，LLM 继续生成。
- 我项目中的 `_think` / `_act` 就是简化版的 Function Calling， myself 做解析而不是让 LLM 输出 JSON。

### 15. LLM 的幻觉问题怎么缓解？
**答（项目相关）：**
- **RAG**：用检索到的真实信息约束生成内容。
- **溯源**：每个观点标注来源，方便人工核查。
- **冲突检测**：发现不一致时提示用户，不强行给出唯一答案。
- **温度控制**：Agent 决策用低 temperature（0.3），保证稳定性。
- **Self-Consistency**：同一个问题多次采样，投票选最一致的答案。

## 三、系统设计 / 场景题

### 16. 如果每天有 10 万用户用这个 Agent，怎么设计后端架构？
**答：**
- **异步化**：调研任务不是瞬时完成的，用 Celery + Redis 做任务队列，用户提交后返回 task_id，前端轮询或 WebSocket 等结果。
- **并发控制**：限制单用户的并发任务数，防止恶意刷接口。
- **缓存**：相同/相似查询缓存结果（Redis），减少 LLM 调用。
- **限流**：API 网关层做 Rate Limiting（如 10 次/分钟/用户）。
- **LLM 路由**：根据任务复杂度选择不同模型（简单任务用便宜模型，复杂任务用强模型），控制成本。
- **监控**：Prometheus + Grafana 监控接口延迟、错误率、Token 消耗。

### 17. 如果要支持"调研我上周问过的问题的最新进展"，怎么设计？
**答：**
- **长期记忆 + 时间感知**：向量数据库存储历史调研的主题、结论、时间戳。
- **增量搜索**：新查询时，先检索历史记录，LLM 判断"这个问题是否已有历史研究"，如果有，在历史基础上补充"最新进展"的搜索词。
- **定时任务**：对热门主题，后台定时自动更新调研报告。

### 18. 如果要求生成 2 万字的长报告，LLM 上下文不够怎么办？
**答：**
- **大纲优先**：先生成大纲，再对每个章节单独调用 LLM 生成内容。
- **Map-Reduce**：每个来源先独立生成摘要（Map），再基于所有摘要生成报告（Reduce）。
- **滑动窗口**：对超长内容分段处理，每段生成后保留关键结论作为下段的上下文前缀。
- **大纲 + 递归扩展**：先写一级标题，再逐层扩展，类似写书的提纲法。

## 四、行为面 / 职业规划

### 19. 你为什么想做 AI Agent 方向？
**答法建议：**
- 从个人兴趣切入：喜欢"让 AI 不只是聊天，而是能真正做事"。
- 结合趋势：Agent 是大模型落地的核心形态，OpenAI、Google、国内大厂都在布局。
- 结合背景：信息与计算科学专业，数学基础 + 编程能力，适合做算法+工程的结合点。
- 强调行动力：为了准备这个方向，自学了 Transformer、RAG、ReAct，并独立做了这个项目。

### 20. 如果实习期间导师给你的任务很杂（写脚本、标注数据），不是核心算法，你怎么办？
**答法建议：**
- 先做好本职工作，建立信任。
- 主动了解团队核心项目，利用业余时间学习相关代码。
- 在合适的时机（如周会、1on1）向 mentor 表达想承担更有挑战的任务的意愿。
- 把杂活也做出亮点：比如写自动化脚本把标注效率提升 3 倍。
